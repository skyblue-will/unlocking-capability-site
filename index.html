Unlocking
Capability

By Will Palmer

The product is not the UI. It never was. User interfaces are cognitive prosthetics — accommodations for how human minds work. What was always underneath: the data model, the business logic, the API. The capability layer.

Agents don't need the accommodation. They mainline into APIs, CLIs, databases. This reveals something that was always true but easy to miss: the interface was never the product. It was the wrapper.

Post-Product

SaaS products bundle three things: capability (what it does), data (what it knows), and interface (how humans access it). Companies competed on how well they executed the bundle.

In a world where agents access capability directly and generate interfaces on demand, the bundle falls apart.

The capability layer is what remains. APIs, data models, business logic. The stuff that actually does things, stripped of any particular human interface.

UIs become ephemeral. Generated when a human needs one. Dissolved when they're done. Not designed once and shipped, but conjured on demand.

This isn't theoretical. I run agents that manage websites, coordinate tasks, and interact with customers — all through capability layers. I still build interfaces, but they're tools for me: shaped to how my brain works, so I can collaborate with the agents effectively. They're not products to be shipped. They're cognitive prosthetics for one person.

And there is no settled "product." As the time between intent and execution collapses, what exists is constantly reshaped — an evolving system of capability layers, personal interfaces, and agent orchestration. You don't ship and maintain. You direct and evolve.

What Building Means Now

If agents write the code, what does a builder actually do?

The shift is from procedural to conceptual. Not "how do I implement this" but "what needs to exist."

A builder needs to know that APIs need authentication, that a feature request might really be an integration problem, that changing pricing logic cascades into billing. They need to understand how a product retains users, why a data model shapes what’s possible, and where a system will break under load. They don’t need to implement these things. They need to know they’re the things to ask for.

This is a confluence of technical and product thinking. The language of product doesn’t disappear — retention, onboarding, pricing, churn — it becomes more important, because the builder now needs both vocabularies. How systems behave, how users behave, and where those two things collide.

Concepts become leverage. Precise instructions produce precise outcomes. Someone without the right mental models can’t access the same capability — not because they lack skill, but because they don’t know what to ask for.

This isn’t about learning a glossary. It’s about understanding why systems work the way they do — technically and commercially. The shift isn’t from code to words. It’s from syntax to judgment.

And that is teachable. Not as a word list, but as mental models.

The capability is democratised.
The governance isn't.
That's the gap.

The Pioneer Problem

Right now, trust is personal. Leadership trusts me because of track record and judgment.

But this doesn't scale. When you want to empower more people, people without that track record, you can't rely on individual judgment.

Pioneers build the path. Others will walk it. The path needs to be safe even for those without the pioneer's footing.

Planning Permission and Building Regulations

When capability is democratised — when anyone with the right mental models can build — the bottleneck moves. It's no longer "can we build this?" It's "should we, and how do we keep it safe?"

This is where human oversight concentrates. Not spread across every task — that's bureaucracy. Concentrated in the governance layer: planning permission, building regulations, and inspection. Three roles, all human, each supported differently by automated systems.

Who can build? That's planning permission. A human with authority looks at what you want to build and says yes or no. It's a judgment call — context, trust, accountability. This stays human. It has to.

How should they build? That's building regulations. Standards that are encoded, testable, enforceable. The API won't accept what isn't permitted. The deployment won't pass if the standards aren't met. But the standards themselves are written by humans — reviewed, debated, updated as the landscape changes. Like a constitution, they need ongoing oversight. No one is letting AI write the regulations. The system enforces them. Humans govern them.

Traditional governance tried to handle both with policy documents. It rarely worked — documents get ignored, and humans can't enforce consistency at scale. Encoding the regulations into the system itself means enforcement is consistent. But enforcement isn't compliance. Compliance requires human inspection — checking the rules are right, that edge cases are caught, that the system is doing what you think it's doing.

But neither planning permission nor building regulations answer the critical question:

Is what was built actually safe? That's inspection. Security review, red teaming, pen testing, auditing. The work that requires adversarial thinking, contextual judgment, and the willingness to try to break what was just built.

Automated systems play a crucial role here — not replacing the humans, but routing work to them. An LLM can audit a codebase and flag what needs attention. Automated checks can enforce the building regulations. But the security layer needs human eyes, because the failure modes are emergent. You can't write a regulation for an attack that hasn't been invented yet.

Not bureaucracy. Not a free-for-all. An agile governance layer where automated systems handle enforcement and routing, and humans concentrate on judgment, standards, and verification.

The AI builds. The human secures. That's the full unlock.

The Trust Gradient

Governance creates a new stratification. Not "technical" versus "non-technical," but a gradient of trust:

Operators work within capability layers others have built. The building regulations protect them; the planning permission was granted to someone else. Builders shape the capability layer — they've earned planning permission for their domain. Architects design the system itself — they set the building regulations.

This isn't a clean hierarchy. Someone might be an architect in one domain and an operator in another. The gradient is continuous and contextual.

When the gap between intent and execution shrinks, incompetence becomes visible quickly. So does competence. The gradient sorts itself faster than any org chart ever could.

The Unlocking

The capability is already there. Agents can both build capability layers and interact with them. The cost is collapsing. The bottleneck isn't technical anymore.

What's missing is the governance. Solve that — who gets planning permission, what the building regulations enforce, who inspects what's built — and you unlock capability for everyone. Not just the pioneers.

The organisations that move first won't be the ones with the best AI. They'll be the ones that build the governance to use it safely.

The system encodes what we've already decided.
Humans decide what's next.
